# CibilBazaar Dialer

A fully offline CRM Dialer: a **Windows desktop app** (Electron + TypeScript)
paired with an **Android app** (Kotlin) over **Bluetooth, USB, or WiFi**, with
automatic reconnect, Excel-based contact management, bulk calling, and a
built-in CRM dashboard. No cloud, no internet dependency — all data stays
between the PC and the paired phone.

See `PROGRESS.md` for the full build log and architecture decisions.

## Fastest Way to Get a Real, Installable .exe and .apk

You don't need Android Studio or Visual Studio installed locally. This repo
includes ready-made GitHub Actions workflows that build both for you in the
cloud:

1. Push this project to a new GitHub repository:
   ```bash
   cd CibilBazaarDialer
   git init
   git add .
   git commit -m "CibilBazaar Dialer"
   git branch -M main
   git remote add origin https://github.com/<your-username>/<your-repo>.git
   git push -u origin main
   ```
2. Open the **Actions** tab on your GitHub repo. Two workflows run
   automatically: **Build Windows Desktop App** and **Build Android APK**
   (or click "Run workflow" to trigger either by hand).
3. When each finishes (5-10 minutes), open the run and scroll to
   **Artifacts**:
   - `CibilBazaarDialer-Windows-Installer` → the real `.exe` — download,
     run it, and it installs like any normal Windows app.
   - `CibilBazaarDialer-Android-Debug-APK` → the real `.apk` — copy it to
     your phone and tap to install (enable "Install from unknown sources"
     the first time).

This works because GitHub's own build servers already have the Windows
toolchain and Android SDK installed — nothing to set up on your side. See
`.github/workflows/build-desktop.yml` and `build-android.yml` for details,
including how to produce a **signed** release APK by adding a keystore as a
repository secret (instructions are in the workflow file's comments).

If you'd rather build locally instead, follow the sections below.

## 1. Desktop App — Build & Run

**Requirements:** Node.js 20.9+ (tested with Node v22 and v24), npm, and on
Windows: Visual Studio Build Tools + Python (for native module compilation —
Bluetooth support).

```bash
cd desktop
npm install
npm run start        # builds + launches Electron in dev
```

`npm install` runs a `postinstall` step (`electron-builder install-app-deps`)
that automatically rebuilds native modules (`better-sqlite3`,
`@serialport/bindings-cpp`) against **Electron's** Node ABI rather than your
system Node's ABI — this is required, since Electron bundles its own Node/V8
and a plain `npm install` alone builds for the wrong ABI, causing a
`NODE_MODULE_VERSION mismatch` crash the first time the app touches SQLite or
a serial port. This step needs internet access (it downloads Electron's
headers) — if it fails behind a corporate firewall/proxy, allow
`electronjs.org` and `github.com`, then re-run `npm install`.

To produce the **Windows installer (.exe)**:

```bash
npm run dist
```

This runs `electron-builder` using the NSIS config already defined in
`package.json`, producing an installer under `desktop/release/`. Supply a
real `build/icon.ico` (256x256) before shipping — `electron-builder` will
use a default icon if it's missing, which is fine for testing.

**Note on Bluetooth:** `bluetoothTransport.ts` requires
`node-bluetooth-serial-port`, a native module. It must be compiled on the
actual Windows build machine (`npm install node-bluetooth-serial-port`) —
native binaries can't be cross-compiled from this development environment.
USB and WiFi transports have no native dependencies and work immediately
everywhere, including for testing the UI/logic on non-Windows machines.

### What each part does
- `src/main/db.ts` — SQLite (contacts, call history, settings)
- `src/main/excelService.ts` — `.xlsx` import/export
- `src/main/wifiTransport.ts`, `bluetoothTransport.ts`, `usbTransport.ts` — the three transports
- `src/main/transportManager.ts` — picks the active transport (USB > Bluetooth > WiFi) and auto-reconnects
- `src/main/callEngine.ts` — single-call and bulk-call state machine
- `src/renderer/` — dark-theme CRM UI (Dashboard, Contacts grid, Call History, Daily Report, Logs, Device Pairing)

## 2. Android App — Build & Run

**Requirements:** Android Studio (Koala or newer), JDK 17.

```bash
cd android
# Open this folder in Android Studio, let Gradle sync, then Run on a device.
```

Or from the command line once the Gradle wrapper is generated by Android
Studio's first sync:

```bash
./gradlew assembleDebug     # debug APK, for testing
./gradlew assembleRelease   # release APK, needs keystore.properties (see below)
```

### Release signing

Generate a keystore once:

```bash
keytool -genkeypair -v -keystore cibilbazaar-release.keystore \
  -alias cibilbazaar -keyalg RSA -keysize 2048 -validity 10000
```

Create `android/keystore.properties` (already git-ignored — never commit
this file):

```properties
storeFile=../cibilbazaar-release.keystore
storePassword=YOUR_STORE_PASSWORD
keyAlias=cibilbazaar
keyPassword=YOUR_KEY_PASSWORD
```

`app/build.gradle.kts` picks this up automatically for `assembleRelease`.

### What each part does
- `protocol/Protocol.kt` — exact mirror of desktop's wire protocol
- `transport/WifiTransport.kt`, `BluetoothTransport.kt`, `UsbTransport.kt`, `TransportManager.kt` — the three transports + priority/reconnect
- `call/DialerController.kt` — places the call via `ACTION_CALL`
- `call/CallStateWatcher.kt` — watches `TelephonyManager` state to compute duration + outcome
- `transport/BridgeService.kt` — foreground service gluing everything together, keeps the connection alive in the background
- `ui/MainActivity.kt`, `ui/PairingActivity.kt` — permissions, status, pairing screens

## 3. First-Time Pairing Workflow

1. Launch the desktop app — it shows a 6-digit **pairing code** on the
   "Device" tab (WiFi) and can scan for OS-paired Bluetooth devices or
   detected USB serial ports.
2. On the phone, open the app → grant all requested permissions → go to
   **Device Pairing**:
   - **WiFi**: type in the 6-digit code (and the desktop's IP only if
     auto-discovery fails on a locked-down network), tap Save & Connect.
   - **Bluetooth**: pair the PC in Android's own Bluetooth settings first
     (standard PIN pairing — this is what secures the channel), then pick
     it from the list.
   - **USB**: plug in the cable, tap Request USB Permission, approve the
     system prompt.
3. The connection status dot on both apps turns green once linked. If the
   link drops, both sides auto-reconnect with the same priority order
   (USB > Bluetooth > WiFi) and exponential backoff — no manual re-pairing
   needed unless the phone or PC actually change.

## 4. Day-to-Day Usage

1. **Import Excel** (Contacts tab) — pick a `.xlsx` with at least a
   `Mobile` column; `Name`, `Company`, `Status`, `Remarks`, `Followup`,
   `Duration`, `Agent` columns are auto-mapped if present.
2. Click **Call** on any row, or use **Start Bulk Calling** to work through
   every `PENDING`/`NO_ANSWER`/`BUSY` contact automatically — pause, resume,
   or stop anytime.
3. The phone auto-opens the dialer for each number; once the call ends, the
   phone reports duration + outcome back, and the desktop grid + Excel file
   update automatically (auto-save, debounced).
4. Track progress in the **Dashboard** (today's KPIs + agent performance),
   **Daily Report** (any date), and **Call History** tabs. **Duplicate**
   mobile numbers are flagged automatically and highlighted in the grid and
   in exported Excel files.

## 5. Fully Offline by Design

Every feature — pairing, calling, Excel sync, reporting — works entirely
over the local Bluetooth/USB/WiFi link between the two devices. No account,
no internet connection, and no third-party server is ever contacted.
